## 🎯 Overview

**S.E.M (SSL Expiry Monitor)** is a web-based ops tool for tracking SSL certificate expiration dates across multiple domains. It features a real-time hero countdown for the most urgent domain, a sorted domain list, and smart status badges — all from a clean, minimalist dashboard.

### ✨ Key Features

- **Hero Countdown**: Live Months / Days / Hours / Mins / Secs timer for the most urgent domain
- **Domain Cards**: Sorted by urgency (most critical first), with progress bars and status badges
- **Smart Status Badges** — calculated live in the browser:
  - ✅ **Active**: > 90 days remaining
  - ⚠️ **Warning**: ≤ 90 days remaining
  - 🔴 **Critical**: ≤ 30 days remaining
  - ❌ **Expired**: Past expiry date
- **Real-time accuracy**: Days remaining and countdown are always current — calculated against `Date.now()`, not frozen snapshot values

## 🔐 SSL Data — How It Works

Domain data is stored in `public/ssl-data.json`. This file is generated by the SSL checker script and served as a static asset.

### Updating Certificate Data

Run this whenever a certificate is **renewed** or a **new domain** is added:

```bash
npm run check-ssl
```

**What it does:**

1. Reads domain list from `domains.config.json`
2. Connects to each host via real TLS handshake (port 443)
3. Extracts live certificate expiry date and issuer from the server
4. Falls back to manually configured dates for wildcard certs (unreachable via base domain)
5. Writes merged results to `public/ssl-data.json`

**When to run it:**
| Trigger | Action needed |
|---------|--------------|
| Certificate renewed | Run `npm run check-ssl` → redeploy |
| New domain added | Add to `domains.config.json` → run script → redeploy |
| Routine (optional) | Can be scheduled as a daily cron job |

> **Note:** The countdown and days-remaining displayed in the UI are **always calculated live in the browser** from the stored `expiryDate`. You only need to re-run the script when an actual expiry date changes (i.e., after renewal).

### Adding a New Domain

Edit `domains.config.json` and add an entry:

```json
{
  "id": "your-domain-com",
  "name": "your-domain.com",
  "checkHost": "your-domain.com",
  "productName": "SSL Certificates",
  "certificateType": "Essential SSL",
  "pricing": "RM159.00",
  "billingCycle": "Annually",
  "fallbackExpiryDate": "2027-01-01T00:00:00+08:00",
  "fallbackIssuer": "Sectigo Limited"
}
```

> `fallbackExpiryDate` and `fallbackIssuer` are used when the live TLS check fails (e.g., wildcard certs, DNS-restricted hosts).

### Automating with GitHub Actions (Optional)

To keep data fresh automatically after deployment, create `.github/workflows/refresh-ssl.yml`:

```yaml
name: Refresh SSL Data
on:
  schedule:
    - cron: '0 2 * * *' # Daily at 2 AM UTC
  workflow_dispatch: # Allow manual trigger

jobs:
  refresh:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm run check-ssl
      - name: Commit updated ssl-data.json
        run: |
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add public/ssl-data.json
          git diff --staged --quiet || git commit -m "chore: refresh ssl-data.json"
          git push
```

Cloudflare Pages will auto-deploy on each commit, keeping everything current.
